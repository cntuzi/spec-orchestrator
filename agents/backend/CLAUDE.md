## Mandatory Rules

### 1. Build Verification
Every code change must compile and pass tests:
```bash
{BUILD_COMMAND}
{TEST_COMMAND}
```
Fix immediately on failure -- do not accumulate errors.

### 2. Build Loop
```
modify -> build -> test -> fix errors -> repeat until all pass
```
Never skip the build step. Never proceed to the next task with a broken build.

### 3. Foreground Execution
- Run all commands in the foreground. No `run_in_background`, no `&`, no chained `sleep`.
- Watch for long-running processes; set appropriate timeouts.

### 4. Worktree Environment
External repo paths (specs, api-doc) are resolved via `.context-resolved.yaml`.
This file is auto-generated by `spec-drive setup` or can be created manually:
```bash
bash {specs_repo}/scripts/resolve-context.sh
```
Resolution order: env var -> local symlink -> git-worktree-discovery -> config hint -> prompt.
If `.context-resolved.yaml` exists, use `repos.specs` and `repos.api-doc` for absolute paths.
Legacy `specs` symlink still works as fallback.

## API Contract Workflow

When implementing a backend task (B{nn}):

1. **Read spec**: `specs/{version}/tasks/backend.md` for task definition.
2. **Read features**: Related Feature YAML `api[]` sections for endpoint contracts.
3. **Implement**: Match the spec exactly (paths, methods, field names, types).
4. **Test**: Write integration tests that verify the contract.
5. **Document**: Update Swagger/OpenAPI definitions.
6. **Verify**: Run tests, verify against acceptance criteria.

---

## Spec-Driven Workflow

When operating under spec-drive orchestration:

1. **Read config**: `.claude/config.yaml` for paths and version.
2. **Check tasks**: `specs/{version}/tasks/backend.md` for assignment and status.
3. **Collect context**: Feature YAML + API contracts.
4. **Lock task**: Update status `pending -> active` and commit.
5. **Implement**: Follow `ai/backend.md` and `ai/api.md` conventions.
6. **Build verify**: Build + test must pass.
7. **Update status**: `active -> done` and commit.
