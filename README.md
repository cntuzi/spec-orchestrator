# Spec Orchestrator

> AI-driven product specification & cross-platform development orchestration.

Turn PRD + Figma + API docs into structured feature specs, then let AI agents autonomously develop your iOS/Android/Web app — with dependency tracking, parallel execution, and change management built in.

## The Problem

AI tools can write code, but software development is more than code:

| Problem | Consequence |
|---------|------------|
| AI doesn't know your tech stack | Generated code is incompatible with your architecture |
| AI doesn't know the design | UI doesn't match Figma — rework cycles |
| AI doesn't know the API contract | Request params are guessed, field names don't match |
| Nobody tracks what's done vs remaining | Progress is tracked by asking around |
| Requirements change mid-sprint | Changed A, but B and C also needed updating — forgot |

## The Solution

A structured **feature spec** layer between PRD and code — a shared language for humans and AI.

```
PRD (natural language for humans)
    ↓  /spec-init
Feature Specs (structured, both human + AI readable)    ← this is the core
    ↓  /spec-drive → AI Workers
Code (for the compiler)
```

One feature spec file consolidates: requirements, Figma references, API contracts, state scenarios, i18n strings, analytics events, and acceptance criteria. AI reads this file and gets complete context — no repeated explanations needed.

## Architecture

Three-layer pipeline: **Generate → Orchestrate → Execute**.

```
/spec-init          Generate spec skeleton from PRD + Figma + Swagger
    ↓
/spec-drive setup   Create version branches + worktrees
/spec-drive next    Analyze dependencies → dispatch parallel AI Workers
    ↓
Worker × N          Autonomous dev loop: Check → Collect → Code → Build → Update
    ↓
/spec-drive done    Version summary + cleanup
```

### Commands

| Command | Purpose |
|---------|---------|
| `/spec-init` | Generate complete spec from PRD + Figma + API |
| `/spec-drive setup` | Initialize version branches |
| `/spec-drive next` | Smart task dispatch with dependency analysis |
| `/spec-drive status` | Real-time cross-platform progress |
| `/spec-drive change` | Track requirement changes + impact analysis |
| `/spec-drive propagate` | Auto-rework for change requests |
| `/spec-drive verify` | Version branch build verification |
| `/spec-next` | View task status, locate next available task |
| `/retro` | Aggregate execution logs into improvement insights |

## Quick Start (5 minutes)

### 1. Clone & Initialize

```bash
git clone https://github.com/anthropics/spec-orchestrator.git  # replace with your fork
cd spec-orchestrator
./scripts/init.sh my-app
```

This creates:

```
my-app/1.0/
├── config.yaml
├── prd/README.md         ← put your PRD here
├── features/             ← auto-generated by /spec-init
├── tasks/
│   ├── shared.md
│   ├── backend.md
│   ├── ios.md
│   └── android.md
├── i18n/strings.md
├── figma-index.md
├── CHANGELOG.md
└── implementation/
```

### 2. Add Your PRD

Place your PRD (Markdown or PDF) in `my-app/1.0/prd/`.

### 3. Generate Specs

```
/spec-init 1.0
```

This reads your PRD + Figma + Swagger and generates:
- Feature YAML for every feature (complete with requirements, API refs, analytics, i18n)
- Platform task plans (iOS + Android)
- Backend API dependency tracking
- Figma page index
- i18n string table

### 4. Start Development

```
/spec-drive setup    # Create version branches
/spec-drive next     # Analyze → dispatch → parallel development
```

## Minimal Adoption (No Tooling Required)

You don't need the full automation to benefit. Start with just Feature YAML files:

```yaml
id: F01
name: User Login
description: Phone number + verification code login

requirements:
  - id: R01
    desc: Enter phone number, tap send verification code
  - id: R02
    desc: Enter code, tap login

state_matrix:
  - id: S01
    name: Empty
    trigger: Open login page
    expected: Phone input focused, login button disabled
  - id: S02
    name: Phone entered
    trigger: Enter 11-digit phone number
    expected: Send code button enabled
```

Just this gives AI tools complete context for the feature. No installation, no commands.

Then incrementally add:
- Task tracking (Markdown tables with status emoji)
- UI constraints (for complex UI features)
- Full automation (spec-init → spec-drive → AI workers)

## Feature YAML — The Core

Each feature gets one YAML file with two categories of fields:

**What (auto-generated from PRD):**
- `requirements` — R01-Rnn requirement list
- `acceptance_criteria` — AC01-ACnn with type (ui/interaction/data)
- `figma.pages[]` — Design page references with node IDs
- `api[]` — Backend endpoint definitions with params/response
- `analytics[]` — Event tracking definitions
- `i18n_ref` — Internationalization string references
- `state_matrix` — Exhaustive state scenario enumeration

**Constraint (human-filled for complex UI):**
- `ui_contract` — Required/forbidden components, key visual tokens
- `delivery_contract` — Tech stack baseline, data field source priority
- `pixel_baseline` — Quantified dimensions (no "looks about right")

Features are classified by `ui_weight`:
- **heavy** — modals, panels, new pages → full constraints needed
- **light** — list items, text changes → state_matrix only
- **logic-only** — pure backend/analytics → no constraints needed

## Change Management

When requirements change mid-sprint:

```
/spec-drive change api /api/endpoint "New field added"
```

This automatically:
1. Traces impact via dependency index (API → Features → Tasks)
2. Creates a Change Record (CR) with propagation checklist
3. Marks affected completed tasks for rework

Then propagate:
```
/spec-drive propagate CR-001
```

AI workers apply only the CR-specific changes, build-verify, and update the checklist.

## Task Lifecycle

```
🔴 Pending → 🟡 In Progress → 🟢 Completed
                                    ↓ CR changes requirement
                                🔵 Rework Needed → 🟡 → 🟢
```

Task files (`tasks/ios.md`, `tasks/android.md`) are the **single source of truth** for status. Dashboard is auto-aggregated, never manually maintained.

## FAQ

**Q: Does this lock me into Claude Code?**
No. The spec files (YAML + Markdown) are tool-agnostic. Any AI tool can read them. The `/spec-*` commands are Claude Code specific but the data format works everywhere.

**Q: Is this for mobile only?**
No. The system supports any platform combination — iOS, Android, Web, backend. Configure your platforms in `.claude/config.yaml`.

**Q: Do I need Figma?**
No. Figma integration is optional. The core value is structured feature specs + task tracking. Figma adds design reference linking.

**Q: How is this different from Jira/Linear?**
Those track "who does what". This tracks "what to build + how to constrain it + what context AI needs". They're complementary.
